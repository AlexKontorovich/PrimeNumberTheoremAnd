# PNT+ Pull Request Style Guide

> **Status: work in progress.** This document is a living record of what
> makes a pull request reviewable in this project. Expect it to grow and be
> revised as new patterns — good and bad — surface in incoming PRs. If you
> hit a case not covered here, or covered ambiguously, note it in your PR
> and the maintainers will fold the lesson back into this file over time.
> See the "How this document evolves" section at the end.

This document supplements [`CONTRIBUTING.md`](CONTRIBUTING.md), which covers
project *process* (claim / disclaim / propose / review). Here we describe
*style*: what a reviewable PR looks like. It applies to human contributors and
to AI-assisted contributors equally — but AI-assisted contributors should read
the "AI-assisted contributions" section, which addresses failure modes
specific to that context.

## Guiding principle

**Reviewer time is the scarcest resource on this project. Every PR should be
designed to make the reviewer's job easier — not the contributor's.**

Modern tools (AI assistants especially) make producing a plausible diff
inexpensive. Understanding, checking, and merging a diff has not gotten
cheaper. If the reviewer has to reconstruct the motivation, chase down
whether the paper actually says what the statement says, verify that a
proof uses the paper's chain of reasoning rather than five bespoke helpers,
and re-run local build checks the contributor didn't bother to run — the
review takes an order of magnitude longer than the writing did, and the
project's throughput is set by that ratio.

Every rule below exists because it makes review faster or higher-signal.
Read each rule with "…so the reviewer doesn't have to" in mind:

- Group thematically related changes into one PR, *so the reviewer doesn't
  have to* re-establish context 12 times.
- State how you verified the change, *so the reviewer doesn't have to*
  guess whether CI is enough or they need to run local checks.
- Use the paper's numbered precursors in your proof, *so the reviewer
  doesn't have to* audit each of your bespoke helpers to check they're
  correct.
- Match the paper's statement verbatim, *so the reviewer doesn't have to*
  read three sources to figure out which one you drifted from.

When in doubt about how to structure a PR, ask: "which shape would take
me the least time to review, if I were on the other side of this?"

## 1. Scope one PR to one motivation

**One PR should have one reason to exist**, phrased as a single sentence you'd
be willing to put in the PR title:

- Good: `fix(FKS2): audit blueprint statements against arXiv:2206.12557` —
  one motivation, may touch many lines across one file.
- Good: `feat(TMEEMT): stub Art06 zeta bounds with blueprint metadata` — one
  file, one motivation.
- Bad: 20 one-line PRs each fixing a different typo in `FioriKadiriSwidinsky.lean`.
  Group them.
- Bad: one PR that both proves a `sorry` in `Kadiri.lean` *and* refactors a
  helper in `LogTables.lean`. Split them.

**Grouping rule of thumb.** If two proposed changes would sensibly appear in
the same paragraph of your commit message, they belong in the same PR. If they
wouldn't, they don't.

**Size guidance** (informal targets):

- A PR under ~500 changed lines is usually easy to review.
- ~500-2000 is manageable if focused on one motivation.
- Larger than that needs a clear "why so big" in the description (e.g. an
  autogenerated table, a mechanical sweep across many similar rows).

**Batch small edits into one PR, not a queue of tiny ones.** If you find 12
blueprint statements that need matching against the paper, open *one* PR
titled `fix(<file>): audit N blueprint statements against <paper>` with all
of them, not twelve one-line PRs — one review pass instead of twelve, and
one CI run instead of twelve. **Excessive numbers of concurrently open PRs
also strain CI** (each PR run compiles the project and the blueprint) and
crowd out other contributors' turnaround time in the shared build queue.

## 2. When in doubt, ask on Zulip before submitting

The project has a Zulip channel:
[`#PrimeNumberTheorem+`](https://leanprover.zulipchat.com/#narrow/channel/423402-PrimeNumberTheorem.2B).
Use it *before* opening the PR whenever any of the following apply:

- You're about to open a large PR (say, more than a few hundred added
  lines, or touching many files) for what sounded on paper like a
  simple task. That mismatch is usually a signal that either the plan
  is wrong or the scope has ballooned; a five-minute Zulip check-in can
  save a full round of review.
- You'd need to introduce a new predicate in `*Definitions.lean`, new
  scaffolding in `*_tables.lean`, or restructure existing definitions.
- You're proposing a new file for a new paper.
- You want to weaken (or strengthen) a `@[blueprint]` `statement` in a
  way that diverges from the paper.
- You're about to submit several related PRs and are unsure whether to
  bundle or split them — describe the plan on Zulip first.
- You're an AI-assisted contributor and your assistant proposed a route
  that doesn't match what you'd expect from reading the paper.

Zulip discussions are cheap; a re-scoped or reverted PR after review is
expensive. When in doubt, a short "I'm thinking of doing X — reasonable?"
message beats a 2000-line surprise.

## 3. Match the paper (for IEANTN paper-formalization work)

The IEANTN subproject formalizes specific published papers. The paper's
statement is the ground truth — the Lean statement should say the same thing.

- **Don't weaken hypotheses to make a proof go through.** If the paper needs
  `x ≥ 10^{19}` and Lean's assumption is `x ≥ 10^{18}`, the theorem is now
  something different — say so in the PR description and open a discussion,
  don't silently ship the divergence.
- **Don't drop `\cite{...}`** when refactoring a statement — the citation is
  the audit trail from the formal statement back to the paper's exact claim.
- **When the paper has an `O^*(…)` or `≤ …` error term, transcribe it
  faithfully**, don't hide it inside an existing predicate that has a
  different shape. (E.g. Trudgian 2014's `N(T)` bound has an additional
  `1/(5T)` term beyond the three-parameter Riemann-von-Mangoldt shape;
  state it directly, don't force-fit the three-parameter template.)

## 4. Reuse over reinvention

Each paper file is a tower of numbered lemmas that the paper's own proof
chain assembles. **The blueprint's `uses := [...]` field is a road map from
each result back to its precursors** — follow it when proving.

Concretely:

- Before writing a new helper, look in the same file for the numbered
  precursors, in the paper's `*Definitions.lean` and `*_tables.lean`
  siblings, and in Mathlib.
- **Antipattern**: closing `sorry` in `lemma_10a` by inventing five
  bespoke sub-lemmas rather than by using `lemma_10_substep` that the
  paper's proof cites. This bloats the file, hides the paper's structure,
  and makes each additional formalization harder because there are more
  ways to close every subsequent `sorry`. **Bloated code is also hard
  to maintain, and it slows down build time for everyone else on the
  project** — every future rebuild of the file pays for your unused
  helpers.
- **`grind`, `polyrith`, and aggressive `simp` are not substitutes** for
  using a stated intermediate result. If the paper writes "by Lemma X.Y",
  the Lean proof should use the Lean statement of that lemma.
- Reusable numeric-bound infrastructure lives in `LogTables.lean`,
  `BKLNW_tables.lean`, `PrimeTables.lean`, `FioriKadiriSwidinsky_tables.lean`,
  `eSHP_tables.lean`, `RosserSchoenfeldPrime_tables.lean`. Reusable
  zeta-side predicates live in `ZetaDefinitions.lean`. Reach for these
  before rolling your own `interval_decide` or your own predicate.
- Reusable summability-of-zeros infrastructure lives in
  `KadiriZeroCounting.lean`. Reach for that too before re-deriving.

**Scope helper lemmas by intended reach.** After you've introduced a
helper, decide honestly whether it's local scaffolding or reusable API:

- If the helper is only used to prove one target result and has no
  plausible use elsewhere, mark it `private`. That keeps it out of the
  file's public namespace and out of the blueprint, so downstream code
  and reviewers aren't tempted to depend on it or wonder what it's for.
- If the helper could plausibly apply to other targets in the project
  — a small numerical fact, a general topology / measure-theory lemma,
  a shape that recurs across papers — leave it public *and* attach a
  `@[blueprint]` node so the blueprint surfaces it as a discoverable
  building block. Public unblueprinted helpers are invisible to
  contributors reading the paper's blueprint page, which defeats the
  purpose of leaving them public.
- The choice isn't about hiding "ugly" internals — it's about the
  project's mental model of what's reusable. Every public unmarked
  helper is one more thing a future contributor has to notice and
  decide whether to use.

**Actively de-duplicate across targets.** If two related tasks would
each need a very similar computation, don't write two independent
proofs. Isolate a single proposition — general enough to cover both
call sites — and use it in both. Two indications that you should be
extracting:

- You've written or generated proof A, and the proof of the next
  target reads like a copy of A with two constants changed. The
  extracted lemma should abstract over those constants.
- Two different sorries have the same rough shape (same tactic chain,
  same auxiliary bounds, same numerical estimate). One proposition
  can usually cover both.

Failing to de-duplicate is a common AI-assistance failure mode: the
tool has full visibility into the goal in front of it but no incentive
to look sideways for similar goals it could share machinery with.
When accepting an AI-suggested proof, ask: "does another sorry in this
file want the same helper?" — and if so, extract before merging.

Reviewers will ask "why not use `<existing lemma>`?" and a satisfying answer
must exist. "The AI suggested it this way" isn't one.

## 5. Compile-time hygiene: shard heavy computations

**Highly computational sub-results should live in separate files with
minimal imports.** Compile time is a shared resource: a rebuild triggered
by editing a proof in one file should not have to re-elaborate 1000
`interval_decide`s or the reduction of a 287-entry numeric table.

Concretely:

- **Numeric tables and certificates go in `*_tables.lean` siblings**
  (`LogTables.lean`, `BKLNW_tables.lean`, `FioriKadiriSwidinsky_tables.lean`,
  `PrimeTables.lean`, `RosserSchoenfeldPrime_tables.lean`, `eSHP_tables.lean`,
  `BKLNW_app_tables.lean`). These files build once and cache; downstream
  paper files consume them as opaque theorems. Follow that pattern for any
  new bulk numeric evidence.
- **Very large certificate corpora get sharded across many small files**
  — the BKLNW Table 10 architecture (~22 files
  `BKLNW_table10_rows_*` + `_dispatch` + `_next` + `_values`) is the
  worked example. Sharding lets Lake compile shards in parallel and lets
  a downstream edit invalidate only the shard(s) it touches, not the
  whole table.
- **Keep imports minimal** in a heavy-computation file. If a table file
  imports the paper file that consumes it (a cycle) or drags in all of
  Mathlib, every downstream edit triggers a needless full rebuild of the
  table. Import only the definitions the table is stated in terms of.
- **Don't inline a 100-row certificate table into a proof file** because
  it's "simpler". It isn't — the next contributor pays the compile cost
  of your convenience every time they touch anything in the paper file.
- If a proof turns out to depend on a new heavy computation, factor the
  computation into a sibling `*_tables.lean` or a new shard, not into
  the proof file. Reviewer time is set partly by CI wall-clock; keep it
  low.

## 6. Blueprint discipline

- **Don't strip `@[blueprint]` metadata** when closing a `sorry` or
  refactoring. The metadata (`statement`, `proof`, `uses`, `latexEnv`, label)
  is the paper-facing side of the formalization; losing it desynchronizes
  the LaTeX blueprint from the Lean file.
- **Update `uses := [...]`** to reflect the actual dependency chain of your
  Lean proof — the blueprint's dependency graph should be honest.
- **When editing a `statement := /-- … -/`, re-verify against the paper.**
  If the change is a divergence from the paper (not a typo fix), open a
  discussion first.
- **LaTeX rules for the `title := …`, `statement := /-- … -/`, and
  `proof := /-- … -/` fields** are documented in this repo's
  [`CLAUDE.md`](CLAUDE.md) — read it if you're editing these. Common
  traps: unicode in a `title`, backticks around a Lean identifier with
  `_` in a `statement`/`proof` field, `\cite[note]{key}` inside a
  `title`. The `lake build :blueprint` step won't catch these; the
  downstream xelatex run will. Verify locally with the xelatex recipe.

## 7. Preserve `sorry`s until you close them

- A PR titled `close sorry in X` should not introduce two new `sorry`s
  in Y and Z as a side effect. If temporarily needed for the refactor,
  say so in the PR body and open a follow-up issue.
- Conversely, don't add sorries to files you didn't intend to touch.

## 8. Verify locally before requesting review

Minimum checks:

1. **`lake build PrimeNumberTheoremAnd.<TouchedModule>`** — clean.
2. **`lake build :blueprint`** — clean (regenerates per-module `.tex`).
3. **For any change to `@[blueprint]` `title`/`statement`/`proof` fields
   or `references.bib`**: run the xelatex check documented in the
   pnt-plus-ieantn skill / `CLAUDE.md` — the LaTeX errors surface only
   there, not in `lake build :blueprint`.
4. **CI green on the PR** before commenting `awaiting-review`. If CI is
   red, either it's your fault (fix it) or a genuine main-was-broken
   situation (say so on the PR).

Do not push and rely on CI as the first check. CI is many minutes per
attempt.

## 9. Commits and PR metadata

**Commit messages**: imperative, ≤ 70 chars, prefixed with a conventional
tag matching the touched file. Examples from the log:
`feat(FKS2): prove theorem_6_2`, `chore(BKLNW): golf ...`,
`feat(CH2): prove prop_2_4_plus`, `fix(TMEEMT): …`.

**PR titles**: same format as the top commit message.

**PR body**: three short sections at minimum.

- **Motivation** (why): one to three sentences. Not "I noticed a typo" —
  state the concrete divergence or gap this PR closes.
- **Scope** (what): a short list of what changed, not one bullet per hunk.
  "Blueprint statements for FKS Thm 2.7 and Prop 3.6 now match arXiv (2.13)"
  is scope. "Modified line 47 of X.lean" is not.
- **Test plan** (verified): the concrete checks you ran locally. Reviewers
  should not have to run these to gain confidence — you should have run
  them and reported the outcome. Untick a box only if it truly wasn't
  applicable; don't leave the boxes unchecked and unexplained.

**Link the issue** if one exists: `Closes #NN` or `Refs #NN`.

**Auto-generated bulleted lists that describe every edit are not a substitute
for the motivation.** If the PR body reads like the output of `git diff`
translated into prose, the reviewer still doesn't know why the PR exists.

## 10. Committing to a branch is cheap; pushing to `main` is not

- Prefer a topic branch off `main`; open a PR from your fork against
  upstream `main`. See `CONTRIBUTING.md` for the mechanics.
- Even maintainers with direct-push rights should route non-trivial or
  bibliography-touching changes through a PR — CI runs *after* a direct
  merge, and a single typo (duplicate bibkey, unicode in a title) can
  break `main` for everyone.
- Don't force-push a shared branch after review has started.
- Rebase over merge before requesting review; keep the branch's history
  linear.

## 11. Draft PRs for design feedback

Open a PR as **draft** if any of the following are true:

- You're introducing a new predicate in `*Definitions.lean` or new
  infrastructure in `*_tables.lean` — reviewers may want to steer the
  shape before you build downstream on it.
- You're refactoring a widely-used definition.
- You're proposing a new file for a new paper (the "bootstrapping"
  workflow in the pnt-plus-ieantn skill).
- You're bulk-editing many files with mechanical similar changes.

Convert to "ready for review" only after CI is green.

## 12. AI-assisted contributions

Contributions produced with the aid of an AI coding assistant (Cursor,
Claude Code, Copilot, Codex, aider, …) are welcome and follow the same
rules as any other contribution — but they should also:

1. **Disclose the tool** in the PR body (e.g. "Made with Cursor", or
   the auto-generated footer produced by these tools). Doesn't need to
   be a manifesto — one line is fine.
2. **Consolidate before submitting**, per §1. If your assistant proposes
   twelve one-line changes to a single file, open one PR, not twelve.
   Reviewer bandwidth doesn't scale with your ease of generating diffs.
3. **Verify the "test plan" you list.** Boxes get checked when you run
   the checks, not when you list them. If the checkboxes in your PR
   body are all unchecked, you have not verified the PR.
4. **Follow §3 (reuse over reinvention) strictly.** Assistants tend to
   close a `sorry` by inventing new machinery rather than by threading
   through the paper's existing proof chain. Before accepting a
   generated proof, check: does it use the numbered precursors the
   paper's own proof cites? If not, ask the assistant to revise.
5. **Understand the diff you submit.** You are the contributor; the AI
   is a tool. If a reviewer asks a basic question about why line X of
   your PR does what it does and the answer is "the AI wrote it", the
   PR isn't ready.
6. **Respect the task board.** The `claim` / `propose #PR` workflow in
   `CONTRIBUTING.md` applies to AI-assisted contributions too. Don't
   push a spontaneous mass-edit that hasn't been claimed via the
   dashboard.
7. **Don't submit AI-generated PRs faster than reviewers can review
   them.** As a rough guideline, don't have more than 3 open AI-assisted
   PRs from you at a time.

## 13. Portions bound for Mathlib

Code intended for upstream Mathlib follows Mathlib's own style manual
(golfing standards, naming conventions, docstring rules, `_root_`
discipline, etc.). See the `mathlib-upstreaming` skill or the
[Mathlib style guide](https://leanprover-community.github.io/contribute/style.html)
for those. **This document does not apply to Mathlib-bound diffs** except
where they overlap (PR scope, motivation, verify-before-review).

If you notice a small helper lemma in this repository that plausibly
belongs upstream, flag it in the PR description; the maintainers can
decide whether to route it via a separate Mathlib PR.

## 14. What NOT to do

- **Don't edit files under `.lake/` or `docbuild/`** — these are build
  artefacts.
- **Don't edit blueprint per-module `.tex` files** under
  `blueprint/src/…` by hand — they're generated from the `@[blueprint]`
  attributes on Lean declarations.
- **Don't include incidental formatting changes** ("while I was there").
  Those go in a separate `chore(<file>): reformat` PR if you must.
- **Don't `git add -A`**. Review what's staged before committing —
  `.lake/` outputs, editor swap files, and stray notes tend to slip in.
- **Don't skip pre-commit hooks or CI gating flags** (`--no-verify`,
  `-c commit.gpgsign=false`) unless explicitly requested by a
  maintainer.

## Antipatterns to watch for in review (reviewer note)

These are the pain-point patterns that motivated this guide. Reviewers may
politely close/reject PRs exhibiting them and request re-submission:

- **PR queue fragmentation.** Many one-line PRs from the same author, of
  the same type, opened within a short window. Ask the author to
  consolidate.
- **Proof-by-bespoke-machinery.** A previously-`sorry`'d numbered lemma
  is now closed but the proof invents new helpers instead of using the
  paper's precursor lemmas. Ask the author to revisit and use the
  existing chain.
- **Diverged statement.** The `statement := /-- … -/` no longer matches
  the paper; usually surfaces as "weakened hypothesis to make proof
  compile". Ask the author to revert the statement and open a discussion.
- **Broken blueprint.** `\cite{…}` with no bib entry, duplicate bibkey,
  unicode in a `title`. Fix in-place if trivial; otherwise revert.
- **Silent stripping of `@[blueprint]`**. Ask the author to restore.
- **Undisclosed AI + unverified checks.** Ask the author to disclose and
  actually run the checks.
- **Heavy computation inlined into a proof file**. Ask the author to
  factor it into a sibling `*_tables.lean` (or a new shard) so downstream
  edits don't pay for the recomputation.
- **Public helper with no `@[blueprint]` tag**. If it's project-scope
  reusable, tag it; if it's local scaffolding, mark it `private`.
  Ask the author to pick.
- **Parallel proofs of parallel computations**. Two nearby sorries
  closed by near-identical proofs (or two similar theorems each with
  their own bespoke helpers) — ask the author to extract the shared
  proposition instead.

## How this document evolves

This guide is not final. Every time a PR surfaces a pattern worth capturing
— a helpful shape, a recurrent antipattern, a review interaction that would
have been shorter with a rule to point at — the maintainers may extend
this file. Contributions to *this file* follow the same rules as any
other: one motivation per PR, verify locally, describe the change in the
PR body.

If you're reviewing a PR and reaching for a critique that doesn't have a
paragraph here yet, that's a signal that the paragraph is worth writing.
Feel free to open a PR against this file with the rule you'd have quoted.
Include a short pointer to the PR (or the class of PRs) that motivated it,
so future readers can see the case behind the rule.

Dating substantive additions in-line (`(added 2026-07-18 after PR #NNNN)`)
is encouraged; it helps future contributors calibrate how load-bearing a
given rule is.
